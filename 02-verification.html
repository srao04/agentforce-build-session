Step 1
Write it as plain English first
Forget variables, conditions, and actions. Describe what this topic should do — in plain language, in order.
Identity check — plain instructions
Your goal is to verify the user's identity before allowing access to order information.

Work through these steps in order:

1.  If you don't have the user's email yet, ask for it:
    "To access order details I will need to verify your identity.
     Could you please provide your email address?"

2.  Once you have an email and haven't run verification yet,
    run the identity verification check using their email address.

3.  If verification succeeds, tell the user:
    "Great news, I have successfully verified your identity!"
    Then move them straight to order lookup.

4.  If verification fails, say:
    "I'm sorry, I wasn't able to verify your identity with the
     information provided. Would you like to try again or create a case?"

    —  If they say Try Again:
       Clear the email, reset isVerified, reset Verification_Run,
       and start from step 1.

    —  If they say Create a Case (or any similar phrasing):
       Record the case type as a verification issue and
       transfer them to case management.
Step 2
What goes where
Every line is either a deterministic guardrail or LLM conversation. The split determines the structure.
Deterministic — variable == value check
LLM — conversation or intent interpretation
Set workflow_step = "Identify User" If already verified → skip to order lookup
If no email and not verified [condition]
Ask the user for their email Capture what they provide
If email + not verified + not yet run [condition] → trigger OrderDetails_Verify_User exactly once → record result in isVerified → set Verification_Run = True
If verified [condition]
Tell the user: "Great news, I have successfully verified your identity!"
→ transition to Order_Status (guaranteed)
If not verified AND verification was run [variable == value check]
Offer retry or case creation
If user says "try again" [LLM interprets intent] User could say: "let me try", "try again", "let's try another email", "retry" — anything → reset email, isVerified, Verification_Run
If user says "create a case" [LLM interprets intent] User could say: "raise a case", "just log it", "open a ticket" — anything → set_CaseType + go_to_Case_Management
Deterministic — before reasoning
Stamps the workflow step for session recovery and fast-paths already-verified users before the LLM is called.
If conditions in before/after_reasoning
Conditions in before/after_reasoning blocks are always system-enforced. Inside reasoning instructions, an if condition can also be LLM-interpreted — the LLM reads the variable and responds accordingly.
LLM — conversation
Asking, wording, and capturing the user's reply are conversational — the LLM handles these naturally.
Deterministic — run exactly once
The verification flow must fire exactly once. The Verification_Run flag guarantees this — the LLM cannot re-run it or skip it.
If condition inside reasoning
This checks isVerified inside reasoning instructions — the LLM reads the variable state and determines what to say next. In before/after_reasoning, this same check would be fully system-enforced.
LLM — conversation
The success message is conversational. The LLM delivers it.
Deterministic — guaranteed transition
Moving to Order_Status after verification is never an LLM choice. It fires automatically in after_reasoning.
Two conditions — both must be true
isVerified == False AND Verification_Run == True. The second condition is the guardrail — it ensures this block only fires after verification has actually been attempted, not on the first turn before the user has even provided an email.
LLM — offer the options
Presenting the retry or case creation choice to the user is conversational — the LLM handles the wording naturally.
LLM-interpreted condition — intent not exact words
There is no exact value to check here. The user could say "let me try", "try again", "let's try another email", "retry" — or anything else that means the same thing. The LLM interprets the intent, then resets the flags and re-asks.
LLM-interpreted condition — intent not exact words
Same pattern — "raise a case", "open a ticket", "just log it" all mean the same thing. The LLM understands intent and selects set_CaseType + go_to_Case_Management regardless of phrasing.
Step 3
Actions
Reasoning actions are tools the LLM picks from based on the conversation. Not everything that runs here is an action.
Actions available for reasoning
set_email → LLM calls this to capture the user's email address set_isVerified → LLM calls this to reset the verified flag on retry set_run_verification → LLM calls this to reset Verification_Run on retry set_CaseType → LLM calls this when user wants to create a case → sets CaseType = "verification_issue" go_to_Case_Management → LLM calls this to transition to Case_Management go_to_order_status → LLM calls this if needed to transition to Order_Status
Not in reasoning actions
OrderDetails_Verify_User This is a topic-level action — it runs deterministically via script conditions. The LLM never decides whether to call it. The if condition triggers it automatically when email is present and the flow hasn't run.
Why these six?
These are all things the LLM needs to choose based on what the user says. Capturing an email, resetting flags on retry, setting a case type — all depend on the conversation going in a particular direction.
Why is OrderDetails_Verify_User excluded?
It is triggered by a script condition — not by an LLM decision. When email is present and Verification_Run is False, the script fires it automatically. The LLM has no say in when or whether it runs.
The key distinction
Reasoning actions = things the LLM selects.
Deterministic calls = things the script triggers.
Both can appear in the same topic — but only one type belongs in the actions list.
Step 4
The structure
The deterministic parts each find their block. The LLM instructions sit in reasoning — plain, undecorated.
before_reasoning
set @variables.workflow_step = "Identify User" if @variables.isVerified == True: transition to @subagent.Order_Status
reasoning instructions
if @variables.UserEmail == "" and @variables.isVerified == False: Ask for email. Say: "To access order details I will need to verify your identity. Could you please provide your email address?" Extract and use {!@actions.set_email} to save it. if @variables.UserEmail != "" and @variables.isVerified == False and @variables.Verification_Run == False: run @actions.OrderDetails_Verify_User with UserEmail = @variables.UserEmail set @variables.isVerified = @outputs.Verified_status set @variables.Verification_Run = True if @variables.isVerified == True: Say: "Great news, I have successfully verified your identity!" if @variables.isVerified == False and @variables.Verification_Run == True: Say: "I'm sorry, I wasn't able to verify your identity. Would you like to try again or create a case?" Try Again → reset email + flags using actions, ask again. Create a case → LLM interprets intent, calls set_CaseType + go_to_Case_Management.
reasoning actions
set_email, set_isVerified, set_run_verification set_CaseType, go_to_Case_Management, go_to_order_status
after_reasoning
if @variables.isVerified == True: transition to @subagent.Order_Status
before_reasoning — always runs first
Stamps the workflow step for session recovery. Fast-paths users who are already verified — no LLM call needed, straight to Order_Status.
Highlighted lines — deterministic
Every purple line is a guardrail the system enforces — if conditions, the verification flow call, and variable assignments. The LLM reasons within these, not around them.
Unhighlighted lines — LLM
What to say, how to handle retry, how to interpret the user's intent for case creation — all natural LLM behaviour. No highlighting needed.
after_reasoning — always runs last
Guarantees the transition to Order_Status fires after the success message — regardless of anything else the LLM said or did.
