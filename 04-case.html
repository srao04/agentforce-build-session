Step 1
Write it as plain English first
This topic handles two completely different journeys — both end in the same place: a case number and a clean reset.
Case management — plain instructions
Your goal is to create a support case and confirm it to the user.
You will always arrive here knowing what type of case to create.

Two journeys lead here — handle each one correctly:

Journey 1 — Verification issue:
    The user could not be verified. Create a verification support case
    using their email address.
    Once created, say:
    "Your verification case has been successfully created.
     Your case number is [case number].
     A member of our support team will reach out to you shortly."

Journey 2 — Expedite or cancel:
    The user wants to expedite or cancel an order. Create an
    expedite/cancel case using the case type, exact order number,
    and user email.
    Once created, say:
    "Your case has been successfully created.
     Your case number is [case number].
     A member of our support team will reach out to you shortly."

After either journey:
    Clear everything — workflow step, case type, all order data,
    case number, and case status — so the agent is ready
    for a completely fresh start.

Step 3
Actions
This is the most deterministic topic in the agent — there are no reasoning actions at all.
No reasoning actions
This topic has no actions available to the LLM. The LLM does not select anything here. It receives the case details and delivers the confirmation message — that is all.
Topic-level actions (script-triggered)
OrderDetails_CreateVerificationCase → triggered when CaseType == "verification_issue" → input: UserEmail → output: CaseNum, CaseStatus OrderDetails_CreateExpediteCancelCase → triggered when CaseType == "Expedite or cancel" → inputs: Case_reason, Order_num, User_email → outputs: CaseCreatedStatus, Case_num
Why no reasoning actions?
By the time the conversation reaches this topic, all decisions have already been made. CaseType is set, ExactOrderNumber is set, UserEmail is set. The script just runs the right flow — there is nothing left for the LLM to choose.
Both flows are deterministic
Each case flow fires because a script condition checks CaseType. The LLM cannot trigger them, reorder them, or skip them. They are fully outside LLM control.
The LLM's only job here
Deliver the confirmation message with the case number. Everything else — which flow runs, what inputs are passed, what gets reset — is handled entirely by the script.
Step 4
The structure
The most deterministic topic in the agent — before_reasoning stamps the step, reasoning runs the flows, after_reasoning resets everything.
before_reasoning
set @variables.workflow_step = "Case Management"
reasoning instructions
if @variables.CaseType == "verification_issue": run @actions.OrderDetails_CreateVerificationCase with user_email = @variables.UserEmail set @variables.casenum = @outputs.CaseNum set @variables.casestatus = @outputs.CaseStatus Say: "Your verification case has been successfully created. Your case number is {!@variables.casenum}. A member of our support team will reach out shortly." if @variables.CaseType == "Expedite or cancel": run @actions.OrderDetails_CreateExpediteCancelCase with Case_reason = @variables.CaseType with Order_num = @variables.ExactOrderNumber with User_email = @variables.UserEmail set @variables.casestatus = @outputs.CaseCreatedStatus set @variables.casenum = @outputs.Case_num Say: "Your case has been successfully created. Your case number is {!@variables.casenum}. A member of our support team will reach out shortly."
after_reasoning — resets 7 variables
set @variables.workflow_step = "" set @variables.CaseType = "" set @variables.InputOrderNumber = "" set @variables.ExactOrderNumber = "" set @variables.OrderDetails = "" set @variables.casenum = "" set @variables.casestatus = ""
before_reasoning — stamps the step
Sets workflow_step = "Case Management" so the topic selector can resume if interrupted mid-case-creation.
Everything is highlighted
Both if conditions, both case flow calls, all input bindings, and all output captures are deterministic. The only unhighlighted lines are the two confirmation messages — those are LLM.
The two confirmation messages — LLM
Delivering the case number to the user is conversational. The LLM reads the stored casenum variable and presents it in the message.
after_reasoning — guaranteed full reset
All seven variables are cleared automatically after the LLM delivers its confirmation. This is the only topic that fully clears everything — it owns the clean exit for the entire agent.
