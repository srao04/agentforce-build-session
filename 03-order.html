Step 1
Write it as plain English first
Describe what this topic should do for a verified user — in plain language, in order.
Order status — plain instructions
Your goal is to look up order details for a verified user.

Work through these steps in order:

1.  If you don't have an order number yet, ask for it:
    "Please provide the order number you would like to look up."
    Capture what the user provides — even a partial number is fine.

2.  Once you have an order number, resolve it to the exact system ID.
    This turns a partial or short number into the full order ID the
    system understands.

3.  If the order does not exist, tell the user and clear the order number
    so they can try again.

4.  Once you have the exact order ID, fetch the full order details —
    status, line items, pricing, shipping, billing, tracking.
    Display everything exactly as returned.

5.  After displaying the order, say:
    "Please let me know if you have any follow up questions about this
     order, or if you would like to expedite your order, cancel your
     order, or look up another order."

    —  Follow-up questions → answer directly from the stored order data.
       Do not re-fetch.

    —  Expedite or cancel → record the case type as "Expedite or cancel"
       and transfer to case management.

    —  Look up another order → clear the order data and ask for the
       new order number.

    —  User is done → say "Let me know if there is anything else I can
       help you with." and clear all order data and the workflow step.
Step 2
What goes where
Two API calls run deterministically. Everything the user says in between is handled by the LLM.
Deterministic — enforced by the system
LLM — handled in conversation
Set workflow_step = "Order Status"
If no order number [condition]
Ask the user for their order number Capture what they provide
If order number given + no exact ID yet [condition] → run GetExactOrderNumber to resolve it → store result in ExactOrderNumber
If order does not exist [condition] → clear InputOrderNumber + ExactOrderNumber
Tell the user the order was not found
If exact order number exists [condition] → run GetOrderDetails to fetch full order → store result in OrderDetails
Display full order details exactly as returned
If order details are loaded [condition]
Answer follow-up questions from stored data No re-fetch needed
Expedite or cancel → LLM interprets intent → set_CaseType + go_to_Case_Management Another order → LLM interprets intent → reset_order_variables, ask again Done → LLM interprets intent → reset_order_variables + set_Workflowstep
Deterministic — before reasoning
Stamps the workflow step so the conversation can resume if interrupted. Runs before the LLM is called.
If conditions inside reasoning
If conditions in before/after_reasoning are always system-enforced. Inside reasoning instructions, an if condition can also be LLM-interpreted — the LLM reads the variable state and decides how to respond.
LLM — conversation
Asking for the order number and capturing the user's response is conversational — the LLM handles it naturally.
Deterministic — resolve the order number
GetExactOrderNumber runs automatically when an order number is present but not yet resolved. The LLM does not decide when this fires.
Deterministic — order not found
If the system returns "Order Does Not Exist", the script clears the variables automatically. The condition is enforced — not interpreted.
LLM — tells the user
Communicating to the user that the order wasn't found is conversational. The LLM delivers the message naturally.
Deterministic — fetch full order
GetOrderDetails runs automatically once ExactOrderNumber is set. The LLM does not decide when or whether to call this.
LLM — display
Displaying the order is conversational. The LLM formats and presents the data exactly as returned.
If conditions inside reasoning
This condition checks OrderDetails — it gates whether the LLM offers follow-up options. Inside reasoning, the LLM reads this variable state and acts accordingly.
LLM — follow-ups from stored data
The LLM reads OrderDetails to answer any follow-up question. No API call needed — data is already in memory.
LLM — interprets user's next intent
The user can say anything — "cancel it", "look up another", "that's all". The LLM interprets the intent and selects the right action.
Step 3
Actions
The LLM picks from these tools based on what the user says. The two API calls are absent — they run deterministically.
Actions available for reasoning
set_InputOrderNumber → LLM calls this to capture the order number the user provided set_CaseType → LLM calls this when user wants to expedite or cancel: sets CaseType = "Expedite or cancel" go_to_Case_Management → LLM calls this to transition to Case_Management reset_order_variables → LLM calls this to clear InputOrderNumber, ExactOrderNumber, and OrderDetails when the user wants another order or is done set_Workflowstep → LLM calls this to clear workflow_step when the user is done
Not in reasoning actions
OrderDetails_GetExactOrderNumber OrderDetails_GetOrderDetails Both are topic-level actions that run deterministically via script conditions. The LLM never decides when to call them — the if conditions trigger them automatically based on variable state.
Why these five?
These are all things the LLM needs to choose based on conversation direction. Capturing the order number, deciding to expedite, deciding to look up another — all depend on what the user says.
Why are the API calls excluded?
GetExactOrderNumber and GetOrderDetails are triggered by script conditions — not LLM decisions. Once the variables reach the right state, the script fires the calls automatically. No LLM input needed.
No after_reasoning needed
Unlike identity check, there is no guaranteed transition out of this topic. The user might ask follow-ups, request another order, or do something unexpected. The LLM handles the exit through actions.
Step 4
The structure
One before_reasoning block, a rich reasoning section, and no after_reasoning — the exit is fully LLM-driven.
before_reasoning
set @variables.workflow_step = "Order Status"
reasoning instructions
if @variables.InputOrderNumber == "": Ask: "Please provide the order number you would like to look up." Extract and use {!@actions.set_InputOrderNumber}. if @variables.InputOrderNumber != "" and @variables.ExactOrderNumber == "": run @actions.OrderDetails_GetExactOrderNumber with Order_number = @variables.InputOrderNumber set @variables.ExactOrderNumber = @outputs.Revised_Order_Number if @variables.ExactOrderNumber == "Order Does Not Exist": Say "Order does not exist" set @variables.InputOrderNumber = "" set @variables.ExactOrderNumber = "" if @variables.ExactOrderNumber != "": run @actions.OrderDetails_GetOrderDetails with Order_Number = @variables.ExactOrderNumber set @variables.OrderDetails = @outputs.Order_Details Display the full order details exactly as returned. if @variables.OrderDetails != "": Say: "Please let me know if you have any follow up questions about this order, or if you would like to expedite your order, cancel your order, or look up another order." Answer follow-ups from {!@variables.OrderDetails}. Expedite/cancel → set_CaseType + go_to_Case_Management Another order → reset_order_variables, ask again Done → reset_order_variables + set_Workflowstep
reasoning actions
set_InputOrderNumber, set_CaseType go_to_Case_Management, reset_order_variables set_Workflowstep
before_reasoning — stamps the step
Sets workflow_step = "Order Status" so the topic selector can resume this conversation if it is interrupted mid-lookup.
Highlighted lines — deterministic
All if conditions, both API call blocks, and the variable assignments are purple. The LLM reasons between these guardrails — it cannot reorder or skip them.
Two sequential API calls
GetExactOrderNumber fires first — it resolves whatever the user typed. GetOrderDetails fires second — it retrieves the full order. Each is gated by the previous variable being set correctly.
Unhighlighted lines — LLM
Asking, displaying, answering follow-ups, interpreting what the user wants next — all natural LLM behaviour. No highlighting needed.
No after_reasoning
This topic has no guaranteed exit. The LLM handles all outbound transitions through actions based on what the user says next.
